# BU01 第一、二阶段固件使用说明

## 1. 当前已实现内容

本固件位于：

`ECE445-main/stm32f103c8t6-led/stm32f103c8t6-led`

核心入口文件：

`Core/Src/main.c`

已经实现三种角色：

| 角色值 | 角色名 | 用途 |
| --- | --- | --- |
| `0` | `DW_ROLE_SELFTEST` | 第一阶段单板自检，验证 SPI、DEV_ID、IRQ、TX |
| `1` | `DW_ROLE_INITIATOR_TAG` | 第二阶段标签端，主动发起 SS-TWR 测距并输出距离 |
| `2` | `DW_ROLE_RESPONDER_ANCHOR` | 第二阶段锚点端，接收 poll 并返回 response |

默认角色是 `0`，也就是单板自检。

## 2. 硬件连接确认

当前工程使用的 BU01/DW1000 引脚如下：

| 功能 | STM32 引脚 |
| --- | --- |
| SPI1_SCK | PA5 |
| SPI1_MISO | PA6 |
| SPI1_MOSI | PA7 |
| DW_CS | PA4 |
| DW_IRQ | PB0 |
| DW_RST | PB12 |
| DW_WAKEUP | PB13 |
| USART1_TX | PA9 |
| USART1_RX | PA10 |

注意事项：

1. `DW_CS` 空闲应为高电平。
2. `DW_IRQ` 已配置为上升沿中断、无上下拉，即浮空输入。
3. 串口波特率为 `115200 8N1`。
4. 程序启动后会读取 DW1000 设备 ID，正常值应为 `0xDECA0130`。

### 3.3 串口期望输出

打开串口助手，参数为：

`115200 8N1`

正常情况下应看到类似输出：

```text
[BU01] Boot
[BU01] Reset DW1000
[BU01] SPI slow init
[BU01] DEV_ID=0xDECA0130
[BU01] Config OK: ch=2 rate=2 preamble=128
[SELFTEST] Phase 1 active
[SELFTEST] Expect: stable DEV_ID=0xDECA0130 and TX_OK every second
[SELFTEST] TX_OK seq=0 irq_total=1 irq_delta=1 status=0x00000080
[SELFTEST] DEV_ID=0xDECA0130
```

验收标准：

1. `DEV_ID` 稳定为 `0xDECA0130`。
2. 每秒输出一次 `TX_OK`。
3. `irq_total` 会递增，说明 DW1000 IRQ 引脚和 STM32 EXTI 可用。

### 3.3 常见异常

如果输出：

```text
[BU01] ERROR: unexpected DEV_ID
```

优先检查：

1. PA4 是否正确接到 BU01 CS。
2. PA5/PA6/PA7 是否正确接到 SPI。
3. BU01 是否 3.3V 供电稳定。
4. PB12 复位脚是否接对。
5. SPI 模式是否被其他代码改动。

如果 `TX_OK` 正常但 `irq_delta=0`：

1. 检查 PB0 是否接到 BU01 IRQ。
2. 确认 IRQ 没有被外部强拉高或强拉低。
3. 确认 GPIO 配置为浮空输入中断。

## 4. 第二阶段：双节点单距离测量

第二阶段需要两块 BU01 开发板：

1. 一块烧录标签端，角色值 `1`。
2. 一块烧录锚点端，角色值 `2`。

```powershell

cmake -S . -B build-tag -G Ninja -DDW_APP_ROLE=1
```

烧录：

`build-tag/stm32f103c8t6-led.hex`

### 4.2 编译锚点端

```powershell

cmake -S . -B build-anchor -G Ninja -DDW_APP_ROLE=2
```

`build-anchor/stm32f103c8t6-led.hex`

### 4.3 标签端串口输出

标签端会输出 CSV 风格测距结果：

```text
[TAG] Phase 2 SS-TWR initiator active
[TAG] CSV: RANGE,seq,distance_cm,status_hex
RANGE,0,103,0x00004000
RANGE,1,101,0x00004000
RANGE,2,102,0x00004000
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `RANGE` | 测距数据行 |
| `seq` | 测距序号 |
| `distance_cm` | 距离，单位 cm |
| `status_hex` | DW1000 状态寄存器 |

### 4.4 锚点端串口输出

锚点端会输出响应日志：

```text
[ANCHOR] Phase 2 SS-TWR responder active
[ANCHOR] RESP_OK seq=0 irq_total=1
[ANCHOR] RESP_OK seq=1 irq_total=2
```

### 4.5 标定实验步骤

建议按以下距离做第一轮标定：

1. 标签和锚点相距 `1 m`，采集 50 到 100 条 `RANGE` 数据。
2. 标签和锚点相距 `3 m`，采集 50 到 100 条 `RANGE` 数据。
3. 标签和锚点相距 `5 m`，采集 50 到 100 条 `RANGE` 数据。

记录每组：

1. 平均值
2. 最大值
3. 最小值
4. 标准差
5. 与真实距离的偏差

如果距离存在固定偏差，优先调整：

```c
#define TX_ANT_DLY 16436
#define RX_ANT_DLY 16436
```

## 5. 当前实现的边界

当前第二阶段采用 SS-TWR，能快速跑通单标签单锚点测距。SS-TWR 对两块板晶振偏差和响应延迟比较敏感，因此标定前的原始误差可能会偏大。后续进入四锚点系统时，建议升级为 DS-TWR 或至少完成天线延迟标定。

当前验收重点：

1. 先确认能稳定通信。
2. 再确认距离随真实距离变化。
3. 最后再追求绝对精度。

### 6.3 近距离临时偏置校正

当前两块板实际相距约 `20-30 cm` 时，未校正的 SS-TWR 原始距离约为 `70-80 cm`。这说明当前天线延迟参数还未完成标定，存在明显固定偏置。

固件中已加入临时偏置：

```c
#define RANGE_BIAS_CM (-46)
```

标签端输出格式已更新为：

```text
RANGE,seq,corrected_cm,raw_cm,status_hex
```

其中：

| 字段 | 含义 |
| --- | --- |
| `corrected_cm` | 加入临时偏置后的距离 |
| `raw_cm` | SS-TWR 原始计算距离 |

本次近距离复测结果：

| 指标 | 数值 |
| --- | --- |
| 样本数 | 33 |
| 校正后平均距离 | 28.79 cm |
| 原始平均距离 | 74.79 cm |

注意：`-46 cm` 只是基于当前近距离摆放的临时偏置，不是最终标定值。正式实验应使用尺子固定 `1 m`、`3 m`、`5 m`，分别记录 `raw_cm`，再拟合最终校正参数。
