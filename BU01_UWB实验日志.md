# BU01 UWB 垒球三维轨迹重建系统实验日志

记录日期：2026-05-10  
项目名称：基于 BU01 UWB 双向测距的垒球三维轨迹重建与落点判定系统

## 1. 项目初始状态

本项目最初的目标是使用 BU01 UWB 模组构建一套低成本、可扩展、可现场部署的垒球轨迹测量系统。BU01 模组基于 Decawave/Qorvo DW1000 超宽带芯片，支持双向测距 TWR 和到达时间差 TDOA。由于垒球场范围较大，锚点之间长期有线同步难度高，而 TDOA 对锚点时钟同步要求很高，因此项目初期确定采用 TWR 作为主实现路线。

系统最初规划为 4 个固定锚点加 1 个球内标签：

- 标签节点安装在球内或临时载体中，主动向锚点发起测距请求。
- 4 个锚点固定在场地周围，按指定时隙响应标签请求。
- 电脑连接其中一个锚点或标签，接收测距结果并完成数据存储、三维重建、轨迹显示和落点判断。

初始阶段的主要风险包括：

- STM32 与 BU01/DW1000 SPI 通信是否稳定。
- DW1000 中断、收发流程是否能跑通。
- TWR 测距误差是否可接受。
- 多锚点场景下如何把数据实时传回电脑。
- UWB 多径、NLOS 和天线姿态变化会不会严重影响三维重建。
- 上位机软件如何将原始距离数据变成可理解的三维轨迹。

## 3. 第二阶段：双节点单距离测量

第二阶段目标是使用 1 个 TAG 和 1 个 Anchor 跑通 TWR 单距离测量。

### 3.1 已完成工作
不同下载器对应的 OpenOCD 配置不同：

- mini HSDAP 使用 CMSIS-DAP 接口。
- ST-LINK V2 使用 ST-LINK 接口。

此外，工程路径中包含中文目录，部分烧录工具在处理中文路径时不稳定，因此后续烧录时采用将 hex 文件复制到临时英文路径的方式，例如：

```text
C:\Temp\uwb_flash
实现了标签端和锚点端两类角色：

- `DW_ROLE_INITIATOR_TAG`：标签端，主动发起测距。
- `DW_ROLE_RESPONDER_ANCHOR`：锚点端，接收 poll 并返回 response。

标签和锚点之间采用单边/改进 TWR 流程，TAG 发起测距，Anchor 响应，TAG 根据发送和接收时间戳计算距离。

实验中先以近距离测试为主，例如 20 cm 至 30 cm、1 m、2.5 m 等距离。后续在 A0 与 TAG 实测距离约 2.5 m 时，原始读数曾稳定在约 3.1 m 至 3.3 m，说明系统存在较明显的系统偏差，需要进行天线延迟或经验偏置修正。

### 3.2 遇到的问题

双节点阶段暴露出几个关键问题：

- 实际距离 2.5 m 时，读数可能在 3.1 m 以上，存在约 60 cm 至 80 cm 系统偏差。
- 天线方向翻转、人体遮挡、狭窄环境反射会让距离明显变大。
- 在 8 m 左右距离时可能稳定测成 10 m 左右，说明远距离或复杂环境下多径和 NLOS 影响严重。
- 串口输出偶尔出现乱码或半帧，特别是在 reset 或重新连接时。

### 3.3 改进方向

针对这些问题，后续确定了三条改进路线：

- 在固件中加入 DW1000 诊断输出，打印接收功率、首径功率、首径指数、power gap。
- 做距离偏置修正，先采用经验上的统一 `-30 cm` 偏置。
- 上位机不直接使用裸距离重建，而是在距离层和坐标层增加异常剔除、插值和平滑。

## 4. 第三阶段：五节点系统与 A0 网关方案

随着硬件增加到 5 块 BU01，系统从双节点扩展到 1 个 TAG 加 4 个锚点。

### 4.1 方案确定

最终确定五节点角色如下：

- A0：锚点 0，同时连接电脑，作为数据回传网关。
- A1：锚点 1，独立供电。
- A2：锚点 2，独立供电。
- A3：锚点 3，独立供电。
- TAG：移动标签，安装在球或临时载体上。

TAG 依次与 A0、A1、A2、A3 测距，得到同一个 sequence 下的四路距离：

```text
d0_cm, d1_cm, d2_cm, d3_cm
```

然后由 A0 通过串口输出统一数据行：

```text
RANGE4D,seq,d0_cm,d1_cm,d2_cm,d3_cm,status_hex,pc_ms,diagnostics...
```

这样电脑只需要连接 A0，就能实时获得 TAG 到四个锚点的距离，不需要同时连接 TAG 或其他锚点。

### 4.2 坐标系方案

实验中使用的锚点坐标配置文件为：

```text
config/anchor_positions.csv
```

当前场地坐标为：

```text
A0 = (0.30, 0.30, 0.40)
A1 = (3.75, 0.00, 0.00)
A2 = (3.75, 3.90, 0.00)
A3 = (0.00, 3.90, 0.00)
```

坐标单位为米。三维重建得到的 TAG 坐标均相对于这个锚点坐标系。

### 4.3 遇到的问题

五节点阶段主要遇到以下问题：

- A0 放在桌面较高位置，A1/A2/A3 放在地面，导致几何结构与初始设想不同，需要重新定义坐标。
- 各锚点烧录角色容易混淆，因此逐步形成 A0/A1/A2/A3/TAG 分别编译和烧录的流程。
- A1/A2/A3 未开机时，A0 与 TAG 的通信仍应存在，但四路完整数据不会全部有效。
- TAG reset 或 A0 reset 后，系统可以重新运行，但串口数据可能出现短暂半帧或乱码。

### 4.4 阶段结果

五节点方案已经跑通：

- A0 能连接电脑并实时输出四锚点距离。
- TAG 可以独立供电运行，不需要长期连接电脑。
- A1/A2/A3 可以独立作为固定锚点参与测距。
- 上位机可以保存实时 CSV 数据。

## 5. 第四阶段：诊断增强与测距质量分析

为了判断 UWB 距离是否可信，固件中加入了 DW1000 诊断信息。

### 5.1 新增诊断字段

CSV 中新增了如下字段：

```text
rxpwr0_cdbm, fppwr0_cdbm, gap0_cdb, fpidx0_q6, rxpacc0
rxpwr1_cdbm, fppwr1_cdbm, gap1_cdb, fpidx1_q6, rxpacc1
rxpwr2_cdbm, fppwr2_cdbm, gap2_cdb, fpidx2_q6, rxpacc2
rxpwr3_cdbm, fppwr3_cdbm, gap3_cdb, fpidx3_q6, rxpacc3
```

其中：

- `rxpwr` 表示接收总功率。
- `fppwr` 表示首径功率。
- `gap = rxpwr - fppwr`，反映总能量与首径能量的差距。
- `fpidx` 表示首径索引。
- `rxpacc` 表示前导累积计数。

### 5.2 对 NLOS 和多径的理解

UWB 芯片测距的关键是判断最早到达的首径信号。如果直达路径被遮挡，芯片可能更容易锁定到反射路径，导致测距变大。

实验观察到：

- 狭窄环境、墙面、桌面、人体附近更容易出现距离变大。
- 天线方向变化会导致部分路径能量变化，距离可能跳变。
- `gap` 较大时，通常说明首径弱、反射能量占比高，多径或 NLOS 风险上升。

### 5.3 改进结果

诊断信息被用于后续三维重建中的加权最小二乘：

- `gap <= 6 dB`：认为该路较可信，权重高。
- `gap >= 14 dB`：认为该路多径风险高，权重降低。
- 中间区间线性降权。

这使得疑似多径的锚点不会在三维重建中把结果强行拉偏。

## 7. 第六阶段：三维重建流程

原始距离数据不能直接作为轨迹显示，需要经过处理链路。

当前处理链路为：

```text
原始 RANGE4D CSV
  -> 距离偏置修正
  -> 距离层异常处理
  -> 三维坐标重建
  -> 3D 坐标层异常过滤
  -> 短异常插值/长异常占位修复
  -> 轻平滑
  -> 图形界面显示
```

对应脚本包括：

```text
tools/uwb_apply_range_bias.py
tools/uwb_filter_ranges.py
tools/uwb_trilateration_3d.py
tools/uwb_filter_positions.py
tools/uwb_process_and_visualize.py
```

### 7.1 距离偏置

目前使用统一经验偏置：

```text
--bias-cm -30
```

即四路距离统一减去 30 cm。它不是严格天线延迟标定，只是根据实验现象做的经验修正。

### 7.2 距离层滤波

距离层主要处理特别大的单路跳变。当前典型参数为：

```text
--max-jump-cm 220
--smooth-window 1
--neighborhood 4
--max-interp-run 3
```

距离层不做强平滑，避免过早把真实运动抹掉。它只负责把明显错误的单路距离短异常进行修复。

### 7.3 三维重建

三维重建基于四个锚点坐标和四路距离：

```text
|P - A0| = d0
|P - A1| = d1
|P - A2| = d2
|P - A3| = d3
```

程序使用加权最小二乘求解三维点，并启用：

```text
z >= 0
```

避免出现地下坐标。

### 7.4 RMS 指标

每个三维点都会输出 `rms_error_m`，表示重建坐标与四路距离之间的一致性：

```text
res_i = |P - Ai| - di
rms = sqrt((res_0^2 + res_1^2 + res_2^2 + res_3^2) / 4)
```

RMS 越小说明四路距离越自洽，RMS 越大说明某些距离互相矛盾，重建结果不可信。

当前常用 RMS 过滤阈值为：

```text
0.30 m
```

## 8. 第七阶段：坐标层滤波与异常修复

随着实测数据增多，发现直接三维重建会出现大量轨迹跳变，尤其是：

- 某些点突然掉到 `z=0`。
- 某些点 RMS 很大但仍被显示。
- 跑动数据后半段较正常，走动数据前半段仍有 Z 轴跳跃。
- 连续几个异常点会让局部轨迹明显偏离。

### 8.1 已实现滤波策略

当前坐标层使用组合策略：

- 绝对 RMS 过滤。
- 局部相对 RMS 过滤。
- 局部位置跳变检测。
- 速度门限检测。
- 加速度门限检测。
- z=0 点过滤或删除。
- 短异常插值。
- 长异常保留占位并插值/保持。
- 中值平滑和 Alpha-Beta 轻平滑。

### 8.2 模式化参数

图形界面中提供不同运动模式：

- 慢走：低速度、低加速度阈值。
- 跑步：中等速度和加速度阈值。
- 快跑：更高速度和加速度阈值。
- 球：允许更高速度和更高加速度。

这样同一套数据处理链可以适配人手移动、跑动测试和后续球体飞行测试。

### 8.3 z=0 处理

界面中加入了两个相关选项：

- 过滤 z=0 点：把 `z <= 0.02 m` 的点标记为异常，然后参与插值修复。
- 删除 z=0 点：把这些点直接从结果中移除。

为了便于追踪处理参数，输出文件名中加入了 z 处理后缀，例如：

```text
zfloor_on
zfloor_drop
```

### 8.4 RMS 阈值可调

图形界面导入原始数据时新增 RMS 阈值输入框，默认：

```text
0.30 m
```

输出文件名中也加入 RMS 后缀，例如：

```text
rms0p30
rms0p15
```

这样可以从文件名直接知道这批处理结果使用了什么过滤强度。

界面开发过程中逐步修正了以下问题：

- 原界面只能一次显示最终轨迹，缺少按 sequence 逐点播放过程。
- sequence 只有 0-255，会循环，导致表格按 sequence 定位时会覆盖或定位到后面的同号点。
- 表格最初只显示最后 200 个点，不适合回看完整轨迹。
- 鼠标左键既要选点又要旋转视角，交互冲突。
- 颜色设置最初没有生效，所有点仍显示白色。

### 9.4 改进结果

最终改为：

- 使用全局 `idx` 作为每行唯一索引。
- 表格显示完整数据。
- sequence 定位采用从当前播放位置向后搜索再回绕。
- 左键只负责选点，右键旋转，中键平移。
- 起点、终点、选中点使用不同颜色。
- 轨迹线使用渐变色，更容易判断运动方向。

## 10. 第九阶段：采样频率提升到 20 Hz

为了获得更平滑的轨迹，项目从较低采样频率提升到 20 Hz。

### 10.1 固件修改

将测距周期改为：

```c
#define RNG_DELAY_MS 50
```

即理论上每 50 ms 一组数据，目标约 20 Hz。

### 10.2 已烧录节点

已经完成 20 Hz 固件烧录：

- A0：网关锚点。
- A1：普通锚点。
- A2：普通锚点。
- A3：普通锚点。
- TAG：移动标签。

### 10.3 实验结果

20 Hz 后数据点更密集，轨迹播放更连续，也给速度和加速度滤波提供了更多时间信息。但同时也出现新的问题：

- 串口 reset 时更容易出现半帧或粘帧。
- 异常点可能连续出现多个，不能只靠单点剔除。
- 采样频率提高并不会自动消除 NLOS 和多径，只是让后端滤波有更多数据可用。

因此后续重点转向“先剔除明显不可信点，再基于可信邻域插值修复”。

## 13. 当前仍存在的问题

系统目前仍处于原型验证阶段，主要问题包括：

- 测距精度仍受天线姿态、遮挡、多径和 NLOS 影响明显。
- 当前 `-30 cm` 是经验偏置，不是严格的天线延迟标定。
- 某些环境下 2 m 至 8 m 的测距误差仍可能达到几十厘米甚至更高。
- reset 或串口重连时可能产生坏行，处理脚本还需要增加 CSV 容错。
- 三维重建依赖锚点坐标准确性，锚点实际摆放误差会直接影响轨迹。
- 当前滤波可以改善显示效果，但过强滤波可能会掩盖真实快速运动。
- 还没有完成真正垒球高速飞行场景下的完整验证。

## 14. 下一步计划

建议后续继续按以下方向推进：

- 增加 CSV 坏行容错，自动跳过字段数量异常的串口粘包行。
- 做更系统的天线延迟标定，不再只使用统一经验偏置。
- 在不同距离点采集标定数据，例如 1 m、2 m、3 m、5 m、8 m。
- 分别测试无遮挡、人体遮挡、墙面反射、天线翻转等工况。
- 将 RMS、gap、速度、加速度等指标统一记录为异常原因，方便后续论文分析。
- 增加落点判定模块，结合场地边界输出是否出界。
- 增加实验截图、轨迹图和典型数据表，用于毕业设计论文和答辩展示。

## 15. 总结

本项目从最初的 BU01 单板连通开始，逐步完成了双节点测距、五节点 TWR 组网、A0 网关回传、诊断增强、实时采集、三维重建、滤波修复和可视化界面。实验过程中暴露出 UWB 原型系统常见的问题，包括系统偏差、多径、NLOS、天线姿态敏感、串口坏帧和三维重建跳变。针对这些问题，项目逐步加入了诊断功率输出、经验偏置、加权重建、RMS 过滤、速度/加速度约束、z=0 处理、插值和平滑等改进。

目前系统已经能够完成从 5 块 BU01 实时测距到电脑端三维轨迹显示的完整链路，具备继续开展垒球运动轨迹实验、落点判定和论文展示的基础。
